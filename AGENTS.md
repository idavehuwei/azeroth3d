# AGENTS · 熔火之心 AI 协作体系

> 本文件定义 Azeroth3D 项目中的 AI 助手角色（Agent）及其职责边界。
> 参考 World of ClaudeCraft 的多角色协作体系，按项目体量精简。
> 更新：2026-07-21

---

## 概述

Azeroth3D 是一个单页 JavaScript 游戏（Three.js + 程序化几何体 + 零外部资源）。
我们的 AI 协作体系不追求 WoC 那种大型仓库的完整角色矩阵，而是按需定义 6 个精干角色，
每个角色只做一件事，做好一件事。

**原则：**
- 每个角色是**只读分析者**或**执行者**，从不混用
- 一个角色对应一个 `Agent` 工具调用，不跨角色复用
- **实现者与审查者必须分离**——写代码的 agent 不审查自己的产出
- 审查者 agent 输出的是「证据清单」，不是「结论」——由主流程决定做什么

---

## 角色清单

| 角色 | 类型 | 触发时机 | 模型建议 |
|------|------|----------|----------|
| `architect` | 只读分析 | 新功能 / 重构前 | 主力模型 |
| `implementer` | 执行 | 方案已定 | 主力模型 |
| `qa-checklist` | 只读分析 | 每次完成变更后 | 主力模型 |
| `test-coverage` | 只读分析 | 新增或修改测试后 | 主力模型 |
| `security-review` | 只读分析 | 涉及输入/存档/联机代码 | 主力模型 |
| `explorer` | 只读搜索 | 大规模查找 | 轻量模型 |

## 角色详述

### 1. architect · 架构师

**职责：** 在实施新功能或重构前，分析现有代码结构，设计方案。

**触发时机：**
- 新 STEP 开始前
- 涉及跨文件修改（3+ 文件）
- 需要决定数据结构和模块拆分

**工作流程：**
1. 读取相关文件，理解当前状态
2. 识别受影响模块和依赖关系
3. 输出方案：`文件清单 → 修改点 → 验收标准`
4. 方案由主流程（而非 implementer）确认后再执行

**约束：** 只读，永不修改文件。

**输出格式：**
```
## 方案：<标题>
### 涉及文件
- `core.js`：新增 xxx 函数
### 修改点
1. 在 core.js 第 N 行后插入 xxx
### 验收
- 全局搜索无魔法数字
- 刷新页面世界位置不变
```

---

### 2. implementer · 执行者

**职责：** 按已确认的方案写入代码。不设计方案，不审查自己。

**触发时机：** architect 方案经主流程确认后。

**工作流程：**
1. 读取方案和必要文件
2. 逐点实施修改
3. 每改完一个点运行 `test_step*.js` 确认绿色
4. 报告完成情况

**约束：**
- 严格按方案执行，不「顺便优化」
- 完成后不自审——交给 qa-checklist

---

### 3. qa-checklist · 质量门禁

**职责：** 变更完成后，逐项检查项目硬性约束是否被违反。

**触发时机：** 每个 STEP 完成、每次提价前。

**检查清单：**
```
[  ] 1. 无魔法数字：所有新数值来自 BALANCE 表
[  ] 2. 世界确定性：world.js / models.js 所有摆放类随机走 srand()，不走 rand()
[  ] 3. 文件头注释：依赖/导出清单已更新
[  ] 4. 无死代码：新增函数被调用
[  ] 5. 实体接口一致：onDeath 回调就位（如果添加了新实体）
[  ] 6. 测试通过：test_step*.js 回归绿色
[  ] 7. 资源纪律：无新增 .png/.glb/.mp3 文件
[  ] 8. 性能：新增粒子/几何体有 dispose() 路径
```

**输出格式：**
```
## QA 报告
[PASS] 1. 无魔法数字：所有数值来自 BALANCE（见 combat.js:24-55）
[PASS] 2. 世界确定性：仅 srand() 用于摆放
[FAIL] 3. 文件头注释：world.js 未更新导出清单
[VERIFY] 6. 测试通过：需要手动运行 test_step01.js
```

**约束：** 只读，永不修改文件。

---

### 4. test-coverage · 测试覆盖率审计

**职责：** 验证测试是否真实保护了所声称的行为，而非仅断言「存在测试」。

**触发时机：**
- 新增或修改 `test_step*.js`
- 声称「有测试覆盖」的变更

**检查要点：**
1. 每个断言是否真的能抓住回归？比如 `ok(hp < 650)` 比 `ok(hp < 650 || hp === 650)` 更严格
2. 边界值是否有覆盖（0 值、最大值、空数组）
3. 死亡实体是否测试了免疫再次受击
4. 浮动随机是否用统计方式验证（1000 次采样而不是单次断言）
5. 新增的死亡回调是否测试了挂接正确

**输出格式：** 报告每个风险点及其置信度。

**约束：** 只读，永不修改文件。

---

### 5. security-review · 安全审计

**职责：** 检查与输入、存档、联机相关的安全风险。

**触发时机：**
- 涉及 `localStorage` 读写
- 涉及用户输入（存档导入、聊天、角色名）
- 涉及 `eval` / `new Function` / `innerHTML` 设置用户控制的文本
- 涉及网络请求

**检查清单：**
```
[  ] 1. innerHTML 是否包含用户可控内容？应当用 textContent
[  ] 2. localStorage 数据是否在读取时验证结构？
[  ] 3. 存档导入是否验证 JSON 结构和类型？
[  ] 4. 是否有 eval / new Function 执行用户输入？
[  ] 5. 联机代码是否防止客户端篡改？
```

**输出格式：** 每个发现的证据和风险等级。

**约束：** 只读，永不修改文件。

---

### 6. explorer · 探索者

**职责：** 在大范围代码库中进行搜索和信息收集。

**触发时机：**
- 需要跨多个文件查找模式
- 不确定某个功能放在哪个文件
- 统计代码量、引用计数

**工作方式：** 使用 `grep` / `find` / `Glob` 等搜索工具，返回结构化的结果。

**输出格式：** 搜索结果摘要，不包含主观分析。

**约束：** 只读，永不修改文件。

## 协作流程

### 标准开发流程（一个 STEP）

```
主流程（你）                  agent
    │                          │
    ├── 读取 plan-merged.md   │
    ├── 理解当前状态          │
    │                          │
    ├──→ architect ──────────→│ 分析当前代码，输出方案
    │←── 方案确认 ────────────│
    │                          │
    ├──→ implementer ────────→│ 按方案实施修改
    │←── 完成报告 ────────────│
    │                          │
    ├──→ qa-checklist ───────→│ 检查硬性约束
    │←── QA 报告 ────────────│
    │                          │
    ├──→ test-coverage ──────→│ 审计测试质量（如有必要）
    │←── 审计报告 ────────────│
    │                          │
    ├── 手动跑回归测试        │
    ├── 更新进度              │
    │                          │
```

### 简单修复流程（bug fix / 小改动）

```
主流程                        agent
    │                          │
    ├──→ implementer ────────→│ 直接修复
    │←── 完成 ────────────────│
    │                          │
    ├──→ qa-checklist ────────→│ 快速检查
    │←── QA 报告 ────────────│
    │                          │
```

### 探索性任务（查找信息）

```
主流程                        agent
    │                          │
    ├──→ explorer ───────────→│ 搜索并返回结果
    │←── 结果 ────────────────│
    │                          │
```

## 模型选择指南

| 任务类型 | 推荐模型 | 原因 |
|----------|----------|------|
| 架构设计 | 主力模型 | 需要深度推理 |
| 代码实现 | 主力模型 | 需要保持代码质量 |
| QA 审查 | 主力模型 | 需要仔细检查细节 |
| 测试审计 | 主力模型 | 需要逻辑推理 |
| 安全审计 | 主力模型 | 不能放过任何点 |
| 大规模搜索 | 轻量模型 | 主要是 grep 操作 |

---

## 与 plan-merged.md 的关系

本文件是**执行层**指南，描述如何做（how）。
`plan-merged.md` 是**策略层**指南，描述做什么（what）。

- 每个 STEP 开始时：主流程读取 plan-merged.md 理解目标
- 然后按本文件流程编排 agent 执行
- 完成后：更新 plan-merged.md 的进度表
